# Requirements Document: S1/S2 × Aspirations-Capabilities Experiments\n\n## Introduction\n\nThis feature implements a systematic experimental framework to test dual-process cognition effects on migration decision-making through aspirations-capabilities theory. The framework builds on Flee's existing S1/S2 implementation (dynamic per-decision switching) and extends it with migration theory concepts. The system will progress through three phases: (1) testing current S1/S2 system, (2) adding aspiration-capability framework, and (3) comprehensive integration testing, culminating in preparation for the South Sudan case study.\n\n## Requirements\n\n### Phase 1: Current S1/S2 System Testing\n\n### Requirement 1\n\n**User Story:** As a researcher, I want to validate the current S1/S2 system behavioral differences, so that I can establish baseline cognitive effects before adding migration theory extensions.\n\n#### Acceptance Criteria\n\n1. WHEN I run experiments with varying conflict levels (0.0 to 1.0) THEN the system SHALL show clear System 2 activation at conflict > 0.6\n2. WHEN agents have connections > 3 AND conflict > 0.6 AND in recovery period THEN the system SHALL activate System 2 with movechance = 1.0\n3. WHEN agents have connections ≤ 3 OR conflict ≤ 0.6 THEN the system SHALL use System 1 with standard movechance calculation\n4. WHEN System 2 is active THEN agents SHALL use pre-calculated routes with enhanced awareness\n5. WHEN System 1 is active THEN agents SHALL calculate routes on-the-fly with standard awareness\n6. WHEN comparing S1 vs S2 decisions THEN the system SHALL show measurable differences in movement timing, route quality, and destination choice\n\n### Requirement 2\n\n**User Story:** As a researcher, I want to test S1/S2 differences across different network topologies, so that I can understand how network structure influences cognitive mode effectiveness.\n\n#### Acceptance Criteria\n\n1. WHEN I create linear topology experiments THEN the system SHALL generate chain networks with configurable length and population distribution\n2. WHEN I create star topology experiments THEN the system SHALL generate hub-and-spoke networks with central hub and multiple destinations\n3. WHEN I create grid topology experiments THEN the system SHALL generate rectangular networks with configurable connectivity density\n4. WHEN I create tree topology experiments THEN the system SHALL generate hierarchical branching networks with configurable depth and branching factor\n5. WHEN running topology comparison experiments THEN the system SHALL measure S1 vs S2 performance differences across network structures\n6. WHEN analyzing topology results THEN the system SHALL identify which topologies amplify or diminish S1/S2 behavioral differences\n\n### Requirement 3\n\n**User Story:** As a researcher, I want to measure information utilization differences between S1 and S2 decisions, so that I can quantify how cognitive modes affect network information processing.\n\n#### Acceptance Criteria\n\n1. WHEN agents share route information THEN the system SHALL track which agents utilize shared information in their decisions\n2. WHEN System 2 is active THEN agents SHALL show higher rates of network information utilization compared to System 1\n3. WHEN measuring information propagation THEN the system SHALL track how quickly destination knowledge spreads through social networks\n4. WHEN analyzing information effects THEN the system SHALL measure correlation between information access and destination choice quality\n5. WHEN System 2 agents have high connectivity THEN they SHALL discover and utilize hidden destination information more effectively than System 1 agents\n6. WHEN generating information utilization reports THEN the system SHALL provide metrics on information discovery rates, sharing patterns, and decision impact\n\n### Phase 2: Aspiration-Capability Framework Integration\n\n### Requirement 4\n\n**User Story:** As a researcher, I want to implement aspiration attributes for agents, so that I can test how S1 vs S2 cognitive modes affect aspiration formation and pursuit.\n\n#### Acceptance Criteria\n\n1. WHEN initializing agents THEN the system SHALL assign aspiration attributes for safety, livelihood, family_reunification, and education\n2. WHEN agents receive information about destinations THEN System 1 SHALL update aspirations using simple heuristic rules\n3. WHEN agents receive information about destinations THEN System 2 SHALL update aspirations using analytical integration of multiple information sources\n4. WHEN agents evaluate destinations THEN the system SHALL calculate aspiration-destination match scores\n5. WHEN tracking aspiration changes THEN the system SHALL log aspiration evolution over time for analysis\n6. WHEN comparing S1 vs S2 aspiration formation THEN the system SHALL show measurable differences in aspiration stability, accuracy, and information integration\n\n### Requirement 5\n\n**User Story:** As a researcher, I want to implement capability attributes for agents, so that I can test how S1 vs S2 cognitive modes affect capability assessment and resource planning.\n\n#### Acceptance Criteria\n\n1. WHEN initializing agents THEN the system SHALL assign capability attributes for financial_resources, social_connections, human_capital, and legal_status\n2. WHEN agents plan journeys THEN System 1 SHALL use simple heuristic capability assessment\n3. WHEN agents plan journeys THEN System 2 SHALL use comprehensive capability analysis with multi-stage resource planning\n4. WHEN agents attempt journeys THEN the system SHALL enforce capability constraints and allow journey failures due to insufficient resources\n5. WHEN tracking capability utilization THEN the system SHALL log resource consumption and planning accuracy\n6. WHEN comparing S1 vs S2 capability assessment THEN the system SHALL show differences in planning accuracy, resource efficiency, and journey success rates\n\n### Requirement 6\n\n**User Story:** As a researcher, I want to implement aspiration-capability matching logic, so that I can test migration theory predictions about when and how agents decide to migrate.\n\n#### Acceptance Criteria\n\n1. WHEN agents evaluate migration decisions THEN the system SHALL calculate migration_intention based on aspiration-capability matching\n2. WHEN System 1 agents assess aspiration-capability match THEN they SHALL use fast, heuristic matching with limited information processing\n3. WHEN System 2 agents assess aspiration-capability match THEN they SHALL use deliberate, analytical matching with comprehensive information integration\n4. WHEN migration_intention exceeds threshold THEN agents SHALL initiate migration planning and execution\n5. WHEN agents reach destinations THEN the system SHALL measure aspiration fulfillment and capability utilization effectiveness\n6. WHEN comparing S1 vs S2 migration decisions THEN the system SHALL show differences in decision timing, destination choice quality, and outcome satisfaction\n\n### Phase 3: Comprehensive Integration Testing\n\n### Requirement 7\n\n**User Story:** As a researcher, I want to run comprehensive experiments testing S1/S2 × aspirations-capabilities interactions, so that I can validate dual-process migration theory predictions.\n\n#### Acceptance Criteria\n\n1. WHEN running aspiration-capability matching experiments THEN the system SHALL test whether S2 agents achieve better aspiration-capability alignment than S1 agents\n2. WHEN running information network experiments THEN the system SHALL test whether S2 agents better utilize social networks for aspiration-relevant information\n3. WHEN running multi-stage journey experiments THEN the system SHALL test whether S2 agents show superior resource planning and journey completion rates\n4. WHEN running dynamic adaptation experiments THEN the system SHALL test whether S2 agents adapt aspirations more effectively to changing conditions\n5. WHEN analyzing experimental results THEN the system SHALL perform statistical significance testing with effect size calculations\n6. WHEN generating experimental reports THEN the system SHALL provide publication-ready results with hypothesis validation summaries\n\n### Requirement 8\n\n**User Story:** As a researcher, I want to create synthetic test scenarios that isolate specific theoretical predictions, so that I can systematically validate dual-process migration theory.\n\n#### Acceptance Criteria\n\n1. WHEN implementing the \"Aspiration-Capability Mismatch\" scenario THEN the system SHALL create destinations with varying aspiration-capability requirements and measure matching accuracy\n2. WHEN implementing the \"Information Cascade\" scenario THEN the system SHALL create hidden information networks and measure discovery rates by cognitive mode\n3. WHEN implementing the \"Resource Planning Challenge\" scenario THEN the system SHALL create multi-stage journeys with resource constraints and measure planning effectiveness\n4. WHEN implementing the \"Dynamic Aspiration Test\" scenario THEN the system SHALL create changing destination conditions and measure adaptation responses\n5. WHEN running scenario comparisons THEN the system SHALL measure relative S1 vs S2 performance across all scenarios\n6. WHEN validating scenario results THEN the system SHALL confirm that theoretical predictions are supported by empirical outcomes\n\n### Requirement 9\n\n**User Story:** As a researcher, I want automated analysis and visualization tools, so that I can efficiently process large experimental datasets and identify patterns.\n\n#### Acceptance Criteria\n\n1. WHEN experiments complete THEN the system SHALL automatically calculate movement timing, route efficiency, and destination quality metrics\n2. WHEN analyzing cognitive differences THEN the system SHALL compute S1 vs S2 performance comparisons with statistical significance tests\n3. WHEN processing aspiration-capability data THEN the system SHALL measure aspiration fulfillment rates, capability utilization efficiency, and migration success rates\n4. WHEN generating visualizations THEN the system SHALL create cognitive state evolution plots, movement pattern comparisons, and parameter sensitivity analyses\n5. WHEN creating reports THEN the system SHALL generate publication-ready figures with proper statistical annotations and effect size reporting\n6. WHEN archiving results THEN the system SHALL ensure full reproducibility with saved configurations, random seeds, and complete metadata\n\n### Requirement 10\n\n**User Story:** As a researcher, I want experimental infrastructure that supports systematic parameter sweeps and topology variations, so that I can explore the full parameter space of dual-process migration theory.\n\n#### Acceptance Criteria\n\n1. WHEN configuring parameter sweeps THEN the system SHALL support systematic variation of conflict thresholds, connectivity levels, and aspiration-capability parameters\n2. WHEN running topology experiments THEN the system SHALL generate and test linear, star, grid, and tree network structures with configurable parameters\n3. WHEN executing parallel experiments THEN the system SHALL manage concurrent simulations with proper resource allocation and error handling\n4. WHEN processing sweep results THEN the system SHALL identify parameter combinations that maximize S1 vs S2 behavioral differences\n5. WHEN analyzing parameter interactions THEN the system SHALL detect phase transitions and critical points in cognitive mode effectiveness\n6. WHEN preparing for case studies THEN the system SHALL provide parameter recommendations and validation criteria for real-world applications\n\n## South Sudan Case Study Preparation\n\n### Requirement 11\n\n**User Story:** As a researcher, I want to prepare the experimental framework for South Sudan case study application, so that I can validate dual-process migration theory with real-world displacement data.\n\n#### Acceptance Criteria\n\n1. WHEN preparing case study infrastructure THEN the system SHALL support real UNHCR displacement data integration and conflict event databases\n2. WHEN calibrating model parameters THEN the system SHALL fit dual-process and aspiration-capability parameters to observed displacement patterns\n3. WHEN validating model predictions THEN the system SHALL compare simulated vs actual displacement outcomes with uncertainty quantification\n4. WHEN generating case study outputs THEN the system SHALL create policy-relevant insights and recommendations based on dual-process theory findings\n5. WHEN documenting methodology THEN the system SHALL provide complete replication materials for independent validation\n6. WHEN creating publication materials THEN the system SHALL combine synthetic experimental results with real-world case study validation\n\n## Technical Requirements\n\n### Requirement 12\n\n**User Story:** As a developer, I want clean integration with existing Flee codebase, so that the experimental framework enhances rather than disrupts core Flee functionality.\n\n#### Acceptance Criteria\n\n1. WHEN extending Person class THEN the system SHALL add aspiration-capability attributes without breaking existing functionality\n2. WHEN modifying decision logic THEN the system SHALL maintain backward compatibility with standard Flee simulations\n3. WHEN adding experimental outputs THEN the system SHALL create new CSV files without modifying existing output formats\n4. WHEN implementing cognitive logging THEN the system SHALL provide configurable logging levels to manage performance impact\n5. WHEN running experiments THEN the system SHALL validate all inputs and provide clear error messages for configuration issues\n6. WHEN integrating with existing tools THEN the system SHALL ensure compatibility with current Flee analysis and visualization scripts\n"
